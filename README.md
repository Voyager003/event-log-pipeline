## Step 1. 이벤트 생성기 작성

### 이벤트 설계

데이터 로깅 관점에서 데이터의 종류는 크게 두 가지로 나눌 수 있는데:

하나는 서비스가 운영되기 위해 필요한 데이터인 `서비스 로그`와 앱이나 웹에서 사용자가 어떤 행동을 하는지 나타내는 `유저 행동 로그`로 구분할 수 있습니다.

과제 배경과 비즈니스 특성을 고려하여 **유저 행동 로그**에 이벤트 위주로 설계하기로 했습니다.

| 이벤트 타입 | 의미 | 선택 이유 |
| --- | --- | --- |
| `LOGIN` | 사용자가 웹 서비스에 로그인 | 로그인 이후 행동을 같은 유저 기준으로 묶기 위한 시작점입니다. |
| `VIEW` | 사용자가 특정 화면 또는 URL에 진입 | 어떤 화면에서 유입과 이탈이 발생하는지 보기 위한 기본 행동입니다. |
| `CLICK` | 사용자가 특정 컴포넌트를 클릭 | 어떤 버튼 또는 UI가 전환 행동을 만드는지 분석하기 위함입니다. |
| `REQUEST` | 사용자의 행동이 서버 API 요청으로 이어짐 | 구매 요청처럼 서버에서 확인 가능한 비즈니스 행동을 기록하기 위함입니다. |
| `LOGOUT` | 사용자가 로그아웃 | 세션 흐름의 종료 지점을 확인하기 위함입니다. |

모든 이벤트는 `eventId`, `eventType`, `source`, `schemaVersion`, `occurredAt`, `userId`, `anonymousId`, `sessionId`, `deviceType`, `trafficSource` 공통 필드를 가집니다. 또한 이벤트 발생 시점의 분석용 유저 프로퍼티로 `membershipLevel`, `lifetimePurchaseCount`, `lifetimePurchaseAmount`, `abTestGroup`을 함께 기록합니다.

## Step 2. 로그 저장

### 저장소 선택

로그 저장소는 RDB인 `PostgreSQL`을 사용합니다. 이유는 생성된 이벤트를 JSON 문자열 하나로 저장하지 않고, 공통 필드와 이벤트별 상세 필드를 컬럼 단위로 분리해 저장하기 좋다고 판단했습니다.

또한 Step 3의 SQL 집계와 Step 5의 Grafana 시각화까지 불편함없이 진행할 수 있을 것이라고 판단했습니다.

### 데이터 구조

- `event_logs`: 모든 이벤트의 공통 envelope와 유저 프로퍼티 스냅샷을 저장
- `view_event_details`: 화면 진입 이벤트의 페이지, URL, referrer, 강의/크리에이터 정보를 저장
- `click_event_details`: 클릭 이벤트의 화면, URL, 컴포넌트 이름과 타입을 저장
- `request_event_details`: 서버 요청 이벤트의 API, 응답 상태, 구매 요청 금액을 저장
- `auth_event_details`: 로그인/로그아웃 이벤트의 인증 제공자와 페이지 URL을 저장

이 구조는 이벤트를 JSON 통째로 저장하지 않고 공통 분석 축과 타입별 상세 필드를 분리하기 위한 설계입니다.

## Step 3. 데이터 집계 분석

유저 행동 로그는 "사용자가 어떤 흐름으로 다음 행동까지 이동했는지"가 중요하다고 판단하여 기본 카운트, 시간 추이, 전환 흐름, 요청 실패율을 함께 작성했습니다.

`event_type_counts.sql`는 사용자는 서비스에서 주로 어떤 행동을 하는가? | 유저 행동 로그 분석의 가장 기본 지표입니다. `VIEW`, `CLICK`, `REQUEST` 비중을 비교하면 사용자가 단순 조회에 머무르는지, 실제 전환 행동까지 이어지는지 큰 흐름을 확인할 수 있습니다.

`hourly_event_trends.sql`는 '사용자는 언제 많이 활동하는가?' 를 분석합니다. D2C 서비스는 시간대별 사용 패턴이 운영과 마케팅 판단에 중요합니다. 특정 시간대에 조회나 요청이 몰리는지 확인하면 프로모션 발송 시간, 운영 모니터링 기준을 정할 수 있습니다.

`purchase_click_to_request_funnel.sql`는 구매 버튼 클릭이 실제 구매 요청까지 이어지는가? | 단순 클릭 수만 보면 전환을 판단하기 어렵습니다. 구매 버튼 클릭 이후 서버 구매 요청까지 이어지는 비율을 보면 UI 클릭과 실제 비즈니스 행동 사이의 이탈을 확인할 수 있습니다.

`request_failure_rate.sql`는 '구매 요청이 서버에서 얼마나 실패하는가?' 유저 행동 로그 중심 설계에서도 `REQUEST`는 서버 응답 상태를 포함합니다. 요청 실패율을 보면 결제/구매 흐름의 기술적 마찰을 확인할 수 있고, 전환율 하락이 사용자 의도 문제인지 시스템 응답 문제인지 구분할 수 있습니다.
