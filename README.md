## Step 1. 이벤트 생성기 작성

### 이벤트 설계

데이터 로깅 관점에서 데이터의 종류는 크게 두 가지로 나눌 수 있는데:

하나는 서비스가 운영되기 위해 필요한 데이터인 `서비스 로그`와 앱이나 웹에서 사용자가 어떤 행동을 하는지 나타내는 `유저 행동 로그`로 구분할 수 있습니다.

과제 배경과 비즈니스 특성을 고려하여 **유저 행동 로그**에 이벤트 위주로 설계하기로 했습니다.

| 이벤트 타입 | 의미 | 선택 이유 |
| --- | --- | --- |
| `LOGIN` | 사용자가 웹 서비스에 로그인 | 로그인 이후 행동을 같은 유저 기준으로 묶기 위한 시작점입니다. |
| `VIEW` | 사용자가 특정 화면 또는 URL에 진입 | 어떤 화면에서 유입과 이탈이 발생하는지 보기 위한 기본 행동입니다. |
| `CLICK` | 사용자가 특정 컴포넌트를 클릭 | 어떤 버튼 또는 UI가 전환 행동을 만드는지 분석하기 위함입니다. |
| `REQUEST` | 사용자의 행동이 서버 API 요청으로 이어짐 | 구매 요청처럼 서버에서 확인 가능한 비즈니스 행동을 기록하기 위함입니다. |
| `LOGOUT` | 사용자가 로그아웃 | 세션 흐름의 종료 지점을 확인하기 위함입니다. |

모든 이벤트는 `eventId`, `eventType`, `source`, `schemaVersion`, `occurredAt`, `userId`, `anonymousId`, `sessionId`, `deviceType`, `trafficSource` 공통 필드를 가집니다. 또한 이벤트 발생 시점의 분석용 유저 프로퍼티로 `membershipLevel`, `lifetimePurchaseCount`, `lifetimePurchaseAmount`, `abTestGroup`을 함께 기록합니다.

### DB 스키마 초안

Step 2 저장 구현을 위해 `src/main/resources/schema.sql`에 PostgreSQL 스키마 초안

- `event_logs`: 모든 이벤트의 공통 envelope와 유저 프로퍼티 스냅샷을 저장합니다.
- `view_event_details`: 화면 진입 이벤트의 페이지, URL, referrer, 강의/크리에이터 정보를 저장합니다.
- `click_event_details`: 클릭 이벤트의 화면, URL, 컴포넌트 이름과 타입을 저장합니다.
- `request_event_details`: 서버 요청 이벤트의 API, 응답 상태, 구매 요청 금액을 저장합니다.
- `auth_event_details`: 로그인/로그아웃 이벤트의 인증 제공자와 페이지 URL을 저장합니다.

이 구조는 이벤트를 JSON 통째로 저장하지 않고 공통 분석 축과 타입별 상세 필드를 분리하기 위한 설계입니다.

### 구현하면서 고민한 점

- 과제 요구사항은 랜덤 이벤트 생성이지만, 테스트와 분석 재현성을 위해 seed 기반 생성 방식을 선택했습니다.
- 실제 서비스에서는 모든 이벤트가 같은 비율로 발생하지 않습니다. 화면 진입과 클릭이 가장 많고 서버 요청과 로그아웃이 상대적으로 적도록 가중치를 두었습니다.
- 이벤트명은 동적으로 만들지 않고 고정된 enum으로 제한했습니다. 그래야 저장, 집계, Grafana 대시보드에서 이벤트 타입 기준 분석이 안정적입니다.
- 유저 프로퍼티는 모든 DB 정보를 넣지 않고, 구매 횟수, 누적 구매 금액, 멤버십, A/B 테스트 그룹처럼 분석에 바로 쓰일 수 있는 스냅샷만 포함했습니다.
